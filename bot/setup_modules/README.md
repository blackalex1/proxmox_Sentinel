# 🛡️ Инструкция по тестированию активной защиты (Aegis IPS)

Эта папка содержит скрипты для автоматического **сквозного (E2E) тестирования** работы системы предотвращения вторжений (IPS) после установки и настройки бота.

В отличие от простых симуляций, тест-скрипт инициирует **реальную подозрительную сетевую активность** (TCP SYN-пакеты на немаршрутизируемый IP-адрес `192.0.2.42` на порт `22`), проверяя полную цепочку работы системы безопасности:
1. Запись события в системные логи ядра (`iptables` -> syslog/journalctl).
2. Чтение логов ботом в реальном времени.
3. Определение процесса нарушителя по порту источника (`ss -atnup`).
4. Принудительное уничтожение процесса (`kill -9`).
5. Отправка тревожного сообщения администраторам в Telegram.

---

## 🚀 Как запустить тестирование

> [!IMPORTANT]
> Для работы тестирования служба бота должна быть предварительно запущена:
> ```bash
> sudo systemctl start proxmox-lxc-bot.service
> ```

Запустите скрипт из-под пользователя `root` (или через `sudo`), находясь в папке `bot`:
```bash
sudo venv/bin/python setup_modules/test_ips.py
```

## 🔍 Что делает скрипт?

1. **Проверка окружения**:
   - Проверяет статус фоновой службы `proxmox-lxc-bot.service`.
   - Проверяет ядерный флаг `net.bridge.bridge-nf-call-iptables` (должен быть равен `1`).
2. **Тестирование Хоста**:
   - Запускает фоновый дочерний процесс, имитирующий исходящее SSH-соединение на sensitive-порт.
   - Проверяет, уничтожил ли бот этот дочерний процесс сигналом `SIGKILL` (сам тестовый скрипт при этом продолжает работать).
3. **Тестирование LXC-контейнеров**:
   - Автоматически опрашивает Proxmox (`pct list`) для поиска запущенных контейнеров.
   - Внутри каждого запущенного контейнера выполняет `pct exec`, запуская там команду соединения (`python3`, `python` или `nc` в зависимости от дистрибутива).
   - Проверяет, был ли процесс прерван IPS-движком.
4. **Тестирование Виртуальных Машин (QEMU)**:
   - Обнаруживает запущенные VM (`qm list`).
   - Выводит статусное сообщение о том, что VM не защищаются IPS-системой (поскольку они изолированы на аппаратном уровне и работают на собственных ядрах, их процессы невозможно контролировать с хоста).
5. **Тестирование удаленных VPS**:
   - Если в `.env` настроен мониторинг внешних VPS, скрипт по SSH запускает там тестовое подключение и проверяет удаленное срабатывание IPS.

## 🛠️ Что делать, если тесты не проходят?

Если в сводной таблице вы видите статус `НЕ СРАБОТАЛО (НЕТ БЛОКИРОВКИ)`:

1. **Проверьте сетевой экран (Firewall) в Proxmox**:
   - Для того чтобы трафик контейнеров перехватывался правилами `iptables` на хосте, в Proxmox VE **обязательно** должен быть включен Firewall для сетевой карты этого контейнера.
   - **Решение**: Откройте Web-интерфейс Proxmox -> Выберите нужный LXC-контейнер -> вкладка **Network** (Сетевые карты) -> выберите `eth0` -> нажмите **Edit** -> Поставьте галочку **Firewall** -> нажмите **OK**.
2. **Проверьте белый список**:
   - Убедитесь, что тестируемый контейнер не внесен в параметр `IPS_LXC_WHITELIST` в файле `config/.env`.
3. **Проверьте статус службы**:
   - Убедитесь, что бот запущен и логирует активность: `sudo journalctl -u proxmox-lxc-bot -f`
